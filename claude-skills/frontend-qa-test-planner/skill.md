이 스킬은 Frontend 소스 코드(React, Vue, Angular, Svelte 등)를 분석하여, 실제 QA 환경에서 즉시 사용 가능한 테스트 항목 체크리스트를 생성합니다. 사용자의 액션(Event)과 시스템의 기대 반응(Expected Result)을 정밀하게 연결하여, QA 담당자가 무엇을 검증해야 하는지 명확히 제시합니다.

## 사용 방법

### 기본 사용법

스킬을 호출할 때 **반드시 분석할 파일을 지정**해야 합니다:

```
/frontend-qa-test-planner @[파일경로]
```

**예시:**
```
/frontend-qa-test-planner @[/frontend/pages/sm/sl/menu-log/index.vue]
```

### 파일 미지정 시 동작

사용자가 파일을 지정하지 않은 경우, **반드시 파일 또는 디렉터리를 지정하도록 요청**합니다:

```
분석할 파일 또는 디렉터리를 지정해주세요.

예시:
- 단일 파일: /frontend-qa-test-planner @[/frontend/pages/sm/sl/menu-log/index.vue]
- 디렉터리: /frontend-qa-test-planner @[/frontend/pages/sm/sl/]
- 다중 파일: /frontend-qa-test-planner @[파일1.vue] @[파일2.vue]
```

**자동 파일 탐색이나 추천은 수행하지 않습니다.** 사용자가 명확하게 파일 경로를 제공할 때까지 분석을 시작하지 않습니다.

### 다중 파일 분석

여러 파일을 한 번에 분석하려면 각 파일을 모두 지정:
```
/frontend-qa-test-planner @[파일1] @[파일2] @[파일3]
```

## 분석 대상

- **주요 파일**: Frontend 컴포넌트 파일 (프로젝트 구조에 관계없이 분석 가능)
  - **Vue / Nuxt** (모든 버전): `*.vue` 파일
    - 페이지 컴포넌트, 레이아웃, 재사용 컴포넌트 등
    - 프로젝트별 경로 예: `/pages/`, `/components/`, `/layouts/`, `/app/`, `/src/`
  - **React / Next.js** (모든 버전): `*.tsx`, `*.jsx` 파일
    - 함수형/클래스형 컴포넌트, 페이지, 레이아웃 등
    - 프로젝트별 경로 예: `/src/`, `/app/`, `/pages/`, `/components/`
  - **Angular** (모든 버전): `*.component.ts`, `*.component.html` 파일
    - 컴포넌트 클래스 및 템플릿
  - **Svelte / SvelteKit** (모든 버전): `*.svelte` 파일
    - 페이지, 컴포넌트, 레이아웃 등
  - **Vanilla JavaScript / TypeScript**: `*.js`, `*.ts` 파일
    - 모든 JavaScript 기반 UI 코드

- **참조 파일**: 해당 컴포넌트에서 import하는 모든 로컬 파일들
  - 하위 컴포넌트
  - Custom Hooks / Composables
  - Utility Functions
  - Type Definitions
  - State Management (Redux, Vuex, Pinia, Zustand 등)
  - API Client / Service Layer

## 분석 프로세스

### 0. 파일 지정 확인 ⚠️

**중요**: 분석을 시작하기 전에 반드시 파일이 지정되었는지 확인:

- ✅ **파일이 지정된 경우**: 즉시 1단계로 진행
- ❌ **파일이 미지정된 경우**: 
  - 사용자에게 파일 또는 디렉터리 경로를 명시적으로 요청
  - 자동 탐색이나 후보 제안을 수행하지 않음
  - 파일 경로가 제공될 때까지 대기

### 1. 주요 파일 읽기
지정된 컴포넌트 파일을 읽습니다.

### 2. 참조 파일 자동 읽기
- 모든 `import` 구문을 파싱합니다.
- 로컬 파일(`~`, `@`, `./`, `../` 등)을 참조하는 모든 파일을 재귀적으로 읽습니다.
- 동적 import, lazy loading 패턴도 추적합니다.

### 3. 컴포넌트 분석
다음 항목들을 중심으로 분석합니다:

**공통 분석 항목:**
- **초기화 로직**: 컴포넌트 마운트 시 실행되는 데이터 로딩, 기본값 설정
- **사용자 이벤트**: 클릭, 변경, 입력, 키보드 이벤트 등 모든 상호작용
- **상태 변화 감지**: 상태 변경에 따른 사이드 이펙트
- **계산된 값**: 파생 데이터의 갱신 조건
- **폼 검증**: 입력값 유효성 검사 로직
- **API 통신**: 데이터 fetching, mutation, 에러 핸들링
- **상태 관리**: 전역/로컬 상태 변화 및 동기화
- **조건부 렌더링**: 조건에 따라 표시/숨김되는 UI 요소
- **엣지 케이스**: 빈 데이터, 오류 상황, 권한, 로딩 상태 등

**프레임워크별 특화 요소:**
- **React**: `useEffect`, `useState`, `useCallback` 등 Hooks 동작
- **Vue**: Lifecycle hooks, `watch`, computed properties, directives
- **Angular**: Lifecycle methods, RxJS subscriptions, Dependency Injection
- **Svelte**: Reactive declarations (`$:`), stores, transitions

## 출력 형식 (정적 테이블)

모든 출력은 반드시 아래 4가지 컬럼을 포함하는 마크다운 테이블 형식으로 제공됩니다:

| 분류 | 점검 항목 | 테스트 데이터 | 기대 결과 |
| :--- | :--- | :--- | :--- |
| **이벤트/기능 단위** | **검증하고자 하는 세부 동작** | **입력값 또는 조작 행동** | **시스템의 최종 상태 및 반응** |

### 다중 파일 분석 시 출력 형식

여러 파일을 분석하는 경우, **페이지(파일)별로 구분하여 출력**합니다:

```markdown
---

## 📄 페이지 1: `/frontend/pages/dashboard/index.vue`

| 분류 | 점검 항목 | 테스트 데이터 | 기대 결과 |
| :--- | :--- | :--- | :--- |
| 초기화면 | ... | ... | ... |
| 버튼클릭 | ... | ... | ... |

---

## 📄 페이지 2: `/frontend/pages/user/profile.vue`

| 분류 | 점검 항목 | 테스트 데이터 | 기대 결과 |
| :--- | :--- | :--- | :--- |
| 초기화면 | ... | ... | ... |
| 폼입력 | ... | ... | ... |

---
```

각 페이지 섹션에는:
- **파일 경로**를 헤더로 명시
- 해당 파일의 **독립적인 테스트 케이스 테이블** 제공
- 페이지 간 **구분선(`---`)** 으로 시각적 분리

### 작성 원칙

- **분류**: 
  - 이벤트 트리거: `초기화면`, `버튼클릭`, `필터변경`, `입력`, `엔터키` 등
  - 기능 단위: `조회`, `등록`, `수정`, `삭제`, `엑셀다운로드`, `페이지네이션`, `정렬` 등
  
- **점검 항목**: 
  - "~~ 확인" 형태로 작성
  - 무엇을 검증하려는지 목적을 명확히 표현
  - 예: "초기 검색 조건 기본값 설정 확인", "필수 입력값 미입력 시 유효성 검사 확인"
  
- **테스트 데이터**: 
  - 테스터가 즉시 실행 가능한 구체적인 값 제시
  - 예: "7일전 - today", "빈 값", "특수문자 포함 문자열", "2페이지 클릭"
  
- **기대 결과**: 
  - 관찰 가능한 UI/상태 변화를 구체적으로 기술
  - 예: "오류 메시지 '필수 항목입니다' 표시", "페이지가 1페이지로 초기화된다", "그리드 데이터가 갱신된다"

## 예시 출력

소스 코드 분석 후 다음과 같은 형식으로 QA 테스트 항목을 제공합니다:

| 분류 | 점검 항목 | 테스트 데이터 | 기대 결과 |
| :--- | :--- | :--- | :--- |
| 초기화면 | 초기 검색 조건 기본값 설정 확인 | 페이지 진입 | 시작일은 10년 전, 종료일은 오늘로 설정된다. |
| 초기화면 | 초기 데이터 조회 수행 확인 | 페이지 진입 | 기본 검색 조건으로 그리드 데이터가 로드된다. |
| 버튼클릭 | 검색 버튼 클릭 시 조회 동작 확인 | 검색 버튼 클릭 | 입력된 조건으로 데이터가 조회되고 그리드가 갱신된다. |
| 엔터키 | 검색어 입력 후 엔터키 입력 시 조회 동작 확인 | 검색어 입력 + Enter 키 | 검색 버튼 클릭과 동일하게 조회가 수행된다. |
| 유효성검사 | 날짜 범위 7일 초과 시 검증 확인 | 시작일: 2026-01-01, 종료일: 2026-01-15 | 경고 메시지 "조회 기간은 최대 7일입니다" 표시 및 조회 차단 |
| 유효성검사 | 필수 선택 항목 미선택 시 검증 확인 | 필터 선택 없이 조회 버튼 클릭 | 오류 메시지 표시 및 조회 차단 |
| 페이지사이즈 | 페이지 당 건수 변경 시 확인 | 20건/50건 선택 | 페이지 사이즈가 변경되고 페이지가 1로 초기화된다. |
| 페이지네이션 | 페이지 번호 클릭 시 목록 갱신 확인 | 2페이지 클릭 | 해당 페이지의 데이터로 그리드가 갱신된다. |
| 정렬 | 컬럼 헤더 클릭 시 정렬 동작 확인 | 날짜 컬럼 헤더 클릭 | 날짜 기준 오름차순/내림차순 정렬이 적용된다. |
| 엑셀다운로드 | 엑셀 다운로드 버튼 클릭 시 동작 확인 | 다운로드 버튼 클릭 | 현재 조회 조건의 전체 데이터가 엑셀 파일로 다운로드된다. |
| 에러처리 | API 호출 실패 시 처리 확인 | 네트워크 오류 발생 | 사용자에게 오류 메시지가 표시되고 로딩 상태가 해제된다. |
| 로딩상태 | 데이터 로딩 중 UI 처리 확인 | API 응답 대기 중 | 로딩 스피너 또는 스켈레톤 UI가 표시된다. |
| 엣지케이스 | 조회 결과 0건인 경우 확인 | 존재하지 않는 조건으로 검색 | "조회된 데이터가 없습니다" 메시지가 그리드에 표시된다. |

## 추가 분석 항목

코드 분석 시 다음 항목들도 함께 식별합니다:

- **접근성(Accessibility)**: 키보드 네비게이션, 스크린 리더 대응
- **반응성(Responsiveness)**: 화면 크기 변경 시 레이아웃 동작
- **성능(Performance)**: 대량 데이터 렌더링, 무한 스크롤 등
- **보안(Security)**: XSS 방지, 입력값 sanitization
- **UX 세부사항**: 로딩 인디케이터, 디바운싱, 낙관적 UI 업데이트 등
